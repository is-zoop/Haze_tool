## 创建入口

进入「开发者中心」后，点击「创建能力」。

创建能力时需要先选择能力类型：

| 类型 | 说明 |
| --- | --- |
| Skill | 用于沉淀提示词、流程、知识文档、分析模板等轻量能力 |
| MCP Server | 用于注册可调用工具、接口服务、自动化任务等可执行能力 |

## 字段填写说明

### 能力名称

能力名称用于在能力市场、开发者中心、审核中心中展示。

推荐写法：

```text
销售日报分析助手
订单状态查询MCP
客服售后话术生成器
商品标题优化Skill
```

不推荐写法：

```text
测试能力
demo
临时工具
我的能力
```

### Slug

Slug 是能力的唯一英文标识，用于系统识别、复制 Prompt、接口调用或后续版本关联。

建议规则：

```text
只使用小写字母、数字和中划线
长度建议 3-30 个字符
不要使用中文、空格、下划线
```

推荐示例：

```text
sales-report-skill
order-query-mcp
customer-service-reply
product-title-optimizer
```

不推荐示例：

```text
SalesReport
销售日报
order_query_mcp
my skill
```

### 能力类型

根据能力实际交付物选择：

| 能力类型 | 应上传工程文件                  |应上传说明文档
| --- |--------------------------|--------|
| Skill | SKILL.md、代码文件等           |README.md、quick_start.md|
| MCP Server | mcp.json、服务代码、依赖文件、测试用例等 |README.md、quick_start.md|

### 业务分类

业务分类用于能力市场筛选。

建议选择最接近能力使用场景的分类，例如：

```text
企业办公
数据分析
电商运营
客服售后
研发工具
财务管理
```

### 能力简介

能力简介建议控制在 1-3 句话，说明能力解决什么问题。

推荐模板：

```text
用于【使用对象】在【业务场景】下，根据【输入内容】生成【输出结果】。
适合处理【典型任务】，可以减少【原本人工成本】。
```

示例：

```text
用于电商运营团队根据每日销售明细生成日报分析，包括 GMV、订单数、退款率和异常波动说明。
适合日常销售复盘、活动期间监控和管理层日报输出。
```

### 标签

标签用于搜索和筛选，建议 2-5 个，使用逗号隔开。

示例：

```text
电商
销售分析
日报
数据清洗
运营复盘
```

## ZIP 包上传

创建能力时需要上传 ZIP 包。

### Skill ZIP 示例

```text
sales-report-skill.zip
├── SKILL.md
├── examples/
│   ├── input.md
│   └── output.md
└── assets/
    └── workflow.png
```

### MCP Server ZIP 示例

```text
order-query-mcp.zip
├── mcp.json         // mcp 说明
├── mcp.yaml         // http模式部署文件
├── package.json     // 安装依赖
├── mcp-test.json   // mcp 测试文件
└── src/
    └── server.ts

```

## 草稿保存

点击「保存」后，能力进入草稿状态。

草稿状态适合：

- 信息还没填完整
- ZIP 包还需要调整
- README 或 SKILL.md 还在修改
- MCP 服务还未准备好测试

草稿不会出现在能力市场中。

## 提交审核

确认信息无误后，点击「提交审核」。

提交审核前建议检查：

- 名称是否清晰
- Slug 是否规范
- 分类是否正确
- 简介是否能说明用途
- ZIP 包是否完整
- 是否包含敏感信息
- Skill 是否包含 `SKILL.md`
- MCP 是否包含 `mcp.json`

## 状态流转

建议按下面状态理解：

```text
草稿 → 审核中 → 审核通过 → 部署中 → 已部署 → 调试通过 → 已发布 → 已下线
```

如果当前系统暂时只显示部分状态，可以按下面方式理解：

| 状态 | 含义 |
| --- | --- |
| 草稿 | 已保存，但未提交审核 |
| 审核中 | 已提交，等待审核 |
| 已拒绝 | 审核未通过，需要修改后重新提交 |
| 已发布 | 已在能力市场可见 |
| 已下线 | 能力已停止对普通用户展示 |

## 修改能力

已保存的能力可以继续编辑。

建议修改规则：

| 阶段 | 建议 |
| --- | --- |
| 草稿 | 可以修改基础信息和 ZIP 包 |
| 审核中 | 不建议频繁修改，避免审核信息不一致 |
| 已发布 | 小改可编辑说明；大改建议创建新版本 |
| 已下线 | 可修改后重新提交或归档 |

## 新版本

当能力已经发布，但需要升级功能时，建议创建新版本。

版本号建议使用语义化格式：

```text
1.0.0
1.1.0
1.2.0
2.0.0
```

版本升级建议：

| 修改类型 | 版本示例 | 说明 |
| --- | --- | --- |
| 文案、说明、示例优化 | 1.0.1 | 不影响能力核心逻辑 |
| 新增小功能 | 1.1.0 | 向下兼容 |
| 重构能力或不兼容变更 | 2.0.0 | 使用方式明显变化 |
