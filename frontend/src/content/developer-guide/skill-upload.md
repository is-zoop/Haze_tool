## 1. 推荐目录结构

请按照以下目录结构组织您的 Skill 工程：

```text
my-skill/
├── SKILL.md         # 必填：能力描述文件，包含元数据、运行场景、权限边界
├── scripts/         # 可选：脚本文件，用于辅助数据处理或简单转换形式
├── references/      # 可选：参考资料，包含专有业务文档或词典库
├── assets/          # 可选：静态资源，如引导图标或提示词附带的规则样式
└── tests/           # 可选：测试用例，用以下行 sandbox 的自动化运行
```

> **提示**：保持根目录结构的整洁性，不仅有利于提高代码的可读性，还能避免平台自动化打包脚本解析出错。

## 2. SKILL.md 编写规范

`SKILL.md` 是 Skill 的核心描述文件，必须包含以下内容：

### 2.1 基本信息
包含 Skill 的名称、版本、作者、归属团队、图标标识以及极简的定位描述。

```yaml
---
name: 销售日报自动分析
version: 1.0.0
author: 数据科学部
department: 数据科学部
tags: [数据分析, 电商运营]
---
```

### 2.2 输入与输出说明
提供业务专用的典型输入字段、期望输出的 Markdown 格式样例。输入越具体越明确，模型输出的效果越稳定。

### 2.3 依赖与权限声明
如果您的 Prompt 需要模型主动调取某些企业资源（例如：“查找王小明最近三天提交的代码”），您必须提供这些资源的依赖声明。

### 2.4 异常处理说明
明确指出当前 Skill 的不适用边界以及在遭遇不可信/缺失数据时的退化兜底话术。

## 3. 权限与安全要求

在上传任何 Skill 到平台前，组织安全部门将进行严格的安全审查：

* **数据出域限制**：禁止通过 Skill 内置提示词将企业内部数据转发或泄露给外部不受信任的接口。
* **写操作兜底防护机制**：涉及数据库或内网表单的修改、写入操作时，必须声明二次用户确认机制。
* **敏感词过滤**：Prompt 中不得含有恶意指令注入（Prompt Injection）防范的弱点漏点。

---

## 4. 测试要求

每次上传或更新 Skill，必须通过以下三种测试：

1. **结构语法核对**：平台校验服务将验证 `SKILL.md` 的 Front Matter 是否完整无缺。
2. **场景完备度评估**：在本地沙箱至少运行 3 组代表性的输入/输出测试，确保流控输出匹配业务设计。
3. **安全红队扫描**：使用内置的安全引擎对 Prompt 针对欺骗提问和越权获取数据进行抵御能力检测。

---

## 5. 发布流程

发布流程主要由以下四个状态环节串联组成：

```text
[本地草稿 Draft] ──> [安全审批 Reviewing] ──> [成功上架 Published] ──> [废弃下线 Offline]
```

1. **草稿**：在本地完成调试，提交并打包（`npm run build`）。
2. **审核**：在开发者页面上报材料，进入组织对应负责人的审批列表。
3. **上架**：审核通过，全员即可在能力市场上检索、收藏并在大模型会话中一键挂载。
