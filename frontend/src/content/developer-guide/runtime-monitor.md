## 运行监控是什么

运行监控用于查看 MCP Server 的运行状态、部署状态和工具调用情况。

适合处理以下问题：

- 服务是否运行
- 最近是否部署成功
- 用户调用是否失败
- 哪个工具耗时较高
- 失败原因是什么
- 是否需要重启服务

## 运行实例

运行实例展示 MCP Server 当前运行状态。

常见状态：

| 状态 | 含义 | 建议操作 |
| --- | --- | --- |
| 运行中 | 服务正常运行 | 可继续观察调用日志 |
| 启动中 | 服务正在启动 | 等待启动完成 |
| 部署中 | 正在部署新版本 | 查看部署记录 |
| 异常 | 服务启动或运行异常 | 查看错误日志 |
| 已停止 | 服务未运行 | 根据需要启动 |

## 常用操作

### 启动

适用于服务已停止，但需要重新提供能力时。

启动前确认：

- 依赖配置完整
- 环境变量已配置
- mcp.json 正确
- mcp.yaml 已配置
- 上一个部署版本可用

### 停止

适用于能力临时停用或发现严重异常时。

停止前建议确认：

- 是否有用户正在使用
- 是否有替代能力
- 是否需要先下线能力

### 重启

适用于服务卡死、配置更新后未生效、临时异常恢复等情况。

重启后建议观察：

- 服务是否进入运行中
- initialize 是否成功
- tools/list 是否正常
- 最近调用是否恢复

## 访问地址

HTTP 类型 MCP Server 通常会展示访问地址。

复制访问地址后可以用于：

- 内部排查
- 接口连通性检查
- 和后端同事确认网关路由
- 对照 mcp.json 中 serverUrl

不要把内部访问地址发送到外部系统或公开文档中。

## 调用日志

调用日志用于查看工具调用记录。

建议关注以下字段：

| 字段 | 说明 |
| --- | --- |
| 调用时间 | 用户或系统发起调用的时间 |
| 工具名称 | 被调用的 MCP Tool |
| 调用结果 | 成功或失败 |
| 耗时 | 工具执行耗时 |
| 错误信息 | 失败时的异常摘要 |

## 调用日志排查方法

### 1. 工具没有被调用

可能原因：

- 工具 description 不清晰
- 用户问题和工具能力不匹配
- 工具未出现在 tools/list 中
- 能力未发布或未启用

处理建议：

- 优化工具 description
- 确认 tools/list 返回工具
- 增加示例用法
- 检查能力状态

### 2. 工具调用失败

可能原因：

- inputSchema 定义错误
- 用户入参不符合要求
- 工具内部代码异常
- 上游 API 异常
- 权限不足

处理建议：

- 查看调用入参
- 对照 inputSchema
- 本地复现工具调用
- 查看服务端日志
- 检查鉴权配置

### 3. 工具调用耗时过高

可能原因：

- 数据库查询慢
- 上游接口慢
- 返回数据过大
- 没有分页或限制条数

处理建议：

- 增加查询条件
- 限制返回条数
- 增加超时处理
- 对慢查询做缓存或异步化

## 部署记录

部署记录用于查看每次部署任务的执行情况。

建议关注：

| 信息 | 说明 |
| --- | --- |
| 部署时间 | 什么时候发起部署 |
| 部署版本 | 部署的是哪个版本 |
| 任务状态 | 成功、失败、执行中 |
| 构建日志 | 镜像构建、依赖安装等信息 |
| 运行日志 | Pod、Service、Gateway 等状态 |
| 错误信息 | 失败原因 |

## 常见部署错误

### 依赖安装失败

示例：

```text
npm install failed
pip install failed
```

处理建议：

- 检查 package.json 或 requirements.txt
- 确认依赖版本是否存在
- 避免使用外网不可访问的私有源
- 如需私有源，说明配置方式

### 启动命令错误

示例：

```text
Cannot find module './dist/server.js'
python: can't open file 'server.py'
```

处理建议：

- 检查 mcp.json 中 command 和 args
- 确认入口文件在 ZIP 包中存在
- Node.js 项目确认是否需要 build
- Python 项目确认文件名是否正确

### 服务启动后退出

示例：

```text
Process exited with code 1
```

处理建议：

- 查看 stderr 日志
- 检查环境变量
- 检查端口、路径、权限
- 在本地用相同命令启动验证

### tools/list 为空

处理建议：

- 确认 MCP Server 注册了 tools 能力
- 确认工具名称和 schema 正常
- 确认 initialize 后可以调用 tools/list
- 查看服务端是否抛出异常

## 运行健康建议

MCP Server 发布后建议定期检查：

- 最近 24 小时失败率
- 平均响应耗时
- 高频失败工具
- 部署失败记录
- 是否存在异常重启
- 是否有用户反馈无法调用

## 异常处理优先级

遇到生产异常时，建议按下面顺序处理：

1. 判断是否影响用户使用。
2. 如果影响范围大，先下线或停止服务。
3. 查看最近部署记录，确认是否由新版本引入。
4. 查看调用日志，定位失败工具和入参。
5. 修复后创建新版本。
6. 重新部署、调试、发布。
7. 在 README 或版本说明中记录修复内容。
