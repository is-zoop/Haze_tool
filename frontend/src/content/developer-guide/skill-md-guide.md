## 概述

`SKILL.md` 是 Skill（技能流）的精髓所在。它不仅是一份让其他开发者阅读的自述文档，还是系统后台自动解析并直接转化为 AI 模型“系统级指令 / 结构化约束”的执行依据。一份糟糕的 `SKILL.md` 会导致自然语言调用大范围失稳，甚至越俎代庖、给出有害或失真的业务建议。

---

## Front Matter 定义规范

每个 `SKILL.md` 文件的最顶端，必须是用两个 `---` 围起来的 YAML 格式定义结构。以下是各基础字段的含义：

```yaml
---
id: "sk_daily_sales"              # 必填。在组织架构中唯一的英文标识符
name: "销售日报分析"              # 必填。向所有终端普通用户展现的对内名称
version: "1.0.0"                  # 必填。语义化版本，每次提交安全审核时建议自增
author: "王小明"                  # 必填。能力维护者的中文真实姓名
department: "电商运营部"          # 必填。对服务提供结算与安全背书的二级部门名
tags: ["电商", "报表清洗"]         # 必填。供服务市场用户搜索的核心推荐标签
capabilities:                     # 可选。能力声明
  - search_grounding              # 支持模型通过内网检索增强可信度
---
```

## 核心章节正格结构

除 Front Matter 外，文档的主体部分需要用 Markdown 进行组织。我们强制约定五个高优先级章节：

### 1. `# 适用场景`
您必须极其清晰地限制模型此项能力施展的典型上下文。

* **优秀写法**：“适用于当运营团队拿到第三方网店昨日销售总表，且昨日交易额数据存在断档或格式混乱时，需要对数据进行对齐与洞察分析的场景。”
* **错误写法**：“大范围的数据清洗。”

### 2. `# 输入与输出说明`
大模型不是占卜家，清晰规范的数据流定义可以让输出准确率提升 `90%` 以上。

* **输入格式示例**：
  ```json
  {
    "channel": "淘系天猫 / 拼多多",
    "rawText": "2026/06/15 数据: 销额 120k..."
  }
  ```
* **输出对齐要求**：指定模型最终必须使用 Markdown 文本对各分析块输出表格，并补充对于波动超过 `20%` 关键指标的警告说明。

### 3. `# 局限与安全边界`
此章节将直接导入模型的负安全防火墙策略。

```text
- 严禁对未经审核的促销方案进行直接上架操作。
- 当昨日营业额与周均值偏差超过 50% 时，模型绝对禁止主动做断言假设，必须提示分析师手工校准。
- 禁止访问或拉取未在 dependency 中报备的企业高密数据库资源。
```

## 实战验证：避免常见格式误区

1. **中文冒号混淆**：
   Front Matter 的 YAML 需要极其严格的 `英文冒号 + 英文空格` 排版。例如 `name: 销售日报自动分析`。若混用 `name：销售日报分析` 会导致平台系统解析器直接崩溃报 500。
2. **多级标题杂乱**：
   请依循一到三级 Markdown 标准标题做排版。请使用 `#`（一级）和 `##`（二级），禁止随心所欲混合粗体和斜体指代重要主线任务。
